# Email Verification Flow

## Overview

This project uses a manual email verification flow.

It does not rely on Laravel's built-in `MustVerifyEmail` trait or the default verification notification flow.
Instead, the application creates the verification link manually, signs it manually, sends the email manually,
and validates the link manually when the user clicks it.

The goal of this implementation is to make the process explicit and easier to learn.

---

## Main Idea

When a user registers, the backend:

1. creates the user
2. dispatches a custom `UserRegistered` event
3. listens for that event
4. generates a verification link
5. signs the link using a secret key
6. sends the link to the user's email

Later, when the user clicks the link, the backend:

1. finds the user by `id`
2. checks that the email hash in the URL still matches the user's current email
3. checks that the link has not expired
4. recomputes the signature
5. compares the new signature with the one from the URL
6. marks the email as verified if everything is valid

---

## Routes

### Verify email

```http
GET /api/auth/email/verify/{id}/{hash}?expires={timestamp}&signature={signature}
```

This route is used when the user clicks the verification link from the email.

### Resend verification email

```http
POST /api/auth/email/send-verification
```

This route is for authenticated users who want to receive the verification email again.

Both routes are throttled to reduce abuse.

## Full Flow

### 1. User registration

When the user sends a registration request, the backend creates the user record and issues the auth tokens.

After that, the backend dispatches a custom event:

```php
event(new UserRegistered($user));
```

This event means:

"A new user has been created. Now run anything that should happen after registration."

In this case, the post-registration action is sending the verification email.

### 2. Event and listener

The event is manually connected to the listener in the auth service provider.

The listener is:

- `SendVerificationEmail`

Its responsibility is simple:

1. receive the `UserRegistered` event
2. extract the user from the event
3. call `EmailVerficationService::send($user)`

This separation is useful because the registration action should only care about registration.
It should not also be responsible for building email links and sending emails.

### 3. Sending the verification email

Inside `EmailVerficationService::send(User $user)`:

1. the service checks whether the user is already verified
2. it calculates the expiration time
3. it generates a verification URL
4. it creates the email
5. it sends the email with `Mail::to(...)->send(...)`

The email itself is rendered using a custom mailable:

- `EmailVerificationMail`

And the HTML content is written in a Blade view:

- `resources/views/emails/auth/verify-email.blade.php`

### 4. Verification link structure

The generated link looks like this:

```text
{APP_URL}/api/auth/email/verify/{id}/{hash}?expires={timestamp}&signature={signature}
```

Example shape:

```text
http://localhost/api/auth/email/verify/5/b76b27e53...?expires=1774215869&signature=ab12cd34...
```

The parts of the URL are:

- `id`: the user ID
- `hash`: `sha1($user->email)`
- `expires`: a Unix timestamp that limits how long the link can be used
- `signature`: an HMAC signature that proves the URL was generated by the server

### 5. Why is the email hash included?

The `hash` part is:

```php
sha1($user->email)
```

This is not the main security layer.
It is mainly used to bind the verification link to the user's current email.

This gives two advantages:

1. the raw email address does not appear in the URL
2. if the user changes their email before using the old link, the old link becomes invalid

So the email hash is a stable fingerprint of the email value.

### 6. How is the signature created?

The signature is created with HMAC:

```php
hash_hmac('sha256', $path . '|' . $expires, $this->signingKey())
```

#### Inputs used in the signature

- `path`
- `expires`
- `APP_KEY` as the secret signing key

#### What is signed?

The signed message is:

```text
{path}|{expires}
```

Where `path` is:

```text
/api/auth/email/verify/{id}/{hash}
```

#### Why sign these values?

Because these are the values that must not be changed:

- the target user
- the target email hash
- the expiration timestamp

If any of these values change, the recomputed signature will be different.

### 7. What is the difference between `sha1` and `HMAC`?

#### `sha1`

`sha1(data)` is a normal hash function.

Properties:

- no secret key
- same input always gives the same output
- anyone can generate it

In this flow, it is used as a fingerprint of the email.

#### `HMAC`

`hash_hmac(...)` is a keyed hash.

Properties:

- uses a secret key
- only the server should know the key
- used to detect tampering

In this flow, HMAC is the real integrity check.

#### Summary

- `sha1(email)` says: "this URL is tied to this email value"
- `HMAC(path|expires, APP_KEY)` says: "the server created this URL and it was not modified"

### 8. How is the signature validated?

When the user clicks the verification link, the backend reaches:

```http
GET /api/auth/email/verify/{id}/{hash}
```

The verification action passes the request data to:

- `EmailVerficationService::verify(...)`

Inside the service, the backend performs these checks:

1. find the user by `id`
2. recompute `sha1($user->email)` and compare it with `{hash}`
3. read `expires` and `signature` from the query string
4. reject the request if the link is expired
5. rebuild the expected signature from the URL data
6. compare the expected signature with the received signature using `hash_equals(...)`

If the comparison succeeds, the link is valid.

If it fails, the link is rejected as invalid or expired.

### 9. Why use `hash_equals(...)`?

The code uses `hash_equals(...)` instead of `===` when comparing hash values and signatures.

This is important because `hash_equals(...)` is designed for secure string comparison.

It helps avoid timing-based attacks where an attacker tries to guess the correct signature by measuring
how long comparisons take.

So:

- `===` is a normal string comparison
- `hash_equals(...)` is the safer choice for cryptographic values

### 10. When is the email marked as verified?

Only after all validation checks pass.

Then the service updates:

```php
$user->forceFill([
    'email_verified_at' => now(),
])->save();
```

That means the database becomes the source of truth.

If `email_verified_at` is not `null`, the user is considered verified.

### 11. Resend verification flow

The resend flow uses the same service method:

```http
POST /api/auth/email/send-verification
```

This route:

1. requires authentication
2. gets the currently authenticated user
3. calls the same `send($user)` method
4. generates a new verification URL
5. sends a new email

So the register flow and resend flow share the same email-generation logic.

### 12. Why this design is useful for learning

This approach is useful because every step is visible:

- the event is manual
- the listener is manual
- the link structure is manual
- the signature is manual
- the email is manual
- the verification logic is manual

This makes it easier to understand what Laravel's built-in verification system is doing behind the scenes.

### 13. Important security notes

#### The link is signed, not encrypted

The link contents are visible.
For example, the user ID and the email hash are visible in the URL.

That is okay.

The goal here is not secrecy.
The goal is integrity:

"The user must not be able to modify the verification data without detection."

#### `APP_KEY` must stay secret

If `APP_KEY` is leaked, an attacker could generate valid signatures.

#### Expiration is required

Without `expires`, the verification link could remain valid forever.

#### Old links can become invalid

If the user's email changes, the old `{hash}` no longer matches the new email value.

That means the old verification link stops working.

### 14. Short summary

This verification flow works like this:

1. user registers
2. backend dispatches `UserRegistered`
3. listener calls `EmailVerficationService::send(...)`
4. service builds `{id}`, `{hash}`, `{expires}`, and `{signature}`
5. service sends the email
6. user clicks the link
7. backend validates user, hash, expiry, and signature
8. backend sets `email_verified_at`

That is the complete manual email verification process in this project.
